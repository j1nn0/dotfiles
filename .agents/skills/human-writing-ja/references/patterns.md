# 避けるべき構造パターン

以下の機械的な構造を見つけたら書き直す。

## 同義反復

同じ内容を少しだけ言い換えて繰り返さない。

### 問題例

```
柔軟な対応が必要です。
そのためには、適応力を高めることが重要です。
変化に対応できる体制を整えることが大切です。
```

3文とも「変化への対応」を言っているだけ。

### 改善

```
状況が変わったら動ける体制を作ります。具体的には、週次で方針を見直す会議を設けます。
```

1つの主張を1回だけ書き、具体策を添える。

### 見分け方

「重要です」「大切です」「必要です」が近くに3つ以上あったら同義反復を疑う。

## 因果関係の欠如

「重要です」で終わって理由がない。

### 問題例

```
セキュリティ対策が重要です。
定期的なアップデートが必要です。
パスワード管理も大切です。
```

なぜ重要なのか、どのくらい重要なのか、わからない。

### 改善

```
先月だけで3件の脆弱性が報告されました。週1回のアップデートを必須にしてください。パスワードは12文字以上、2要素認証を義務化します。
```

理由（数字、事実）→ 具体的なアクション の順番で書く。

## 文の長さの均一化

すべての文が同じくらいの長さだと、抑揚がなく平板になる。

### 問題例

```
この機能は便利です。設定も簡単です。すぐに使えます。効果も高いです。
```

すべて8〜10文字。読んでいて眠くなる。

### 改善（ですます調を維持する場合）

```
便利です。設定は3クリックで終わります。使い始めて1週間で、問い合わせ対応の時間が半分になりました。
```

### 改善（だ/である調に変える場合）

```
設定は3クリックで終わる。便利だ。使い始めて1週間で、問い合わせ対応の時間が半分になった。
```

短い文（6文字）と長い文（30文字以上）を混ぜる。文体はどちらでもよい。

## 箇条書きの過剰な整形

同じ文字数、同じ語尾、完璧に構造化された箇条書きは「説明書感」が出る。

### 問題例

```
機能の特徴は以下の通りです。

- 高速な処理が可能です
- 安全な通信を実現します
- 簡単な操作で利用できます
- 柔軟なカスタマイズに対応します
```

4項目すべてが「〜します」「〜です」で終わり、文字数もほぼ同じ。

### 改善

```
処理速度は従来の3倍です。通信はTLS 1.3で暗号化しています。操作画面は3ステップで完結します。
```

箇条書きをやめて文章にし、語尾を変える（「可能です」→「3倍です」、「実現します」→「しています」、「利用できます」→「完結します」）。具体的な数字も入れる。

## 予告と要約の冗長さ

### 問題例

```
本記事では、タスク管理の方法について解説します。まず背景を説明し、次に具体的な方法を紹介し、最後にまとめます。

（本文）

まとめ
本記事では、タスク管理の3つの方法について解説しました。これらの方法を活用することで、業務効率を向上させることができます。
```

予告も要約も読者は求めていない。「本記事では」が2回出てくる時点でAI臭い。

### 改善

本題から始め、本題で終わる。予告文は削除。要約は本当に必要な場合だけ、新しい情報を添えて書く。

## 定型セクション構成

### 問題例

```
## 背景
## 目的
## 方法
## 結果
## 考察
## まとめ
```

または

```
## メリット
## デメリット
## まとめ
```

学術論文や報告書のフォーマットをそのまま使っている。内容に合わせて取捨選択すべき。

### 改善

必要なセクションだけ残す。「まとめ」は本文で十分伝わっているなら削除。

## 二項対立の多用

### 問題例

```
量だけでなく質も重要です。
過去に囚われず、未来を見据えましょう。
理論と実践の両輪が必要です。
個人の成長だけでなく、組織の成長も考慮すべきです。
```

1つの文書に4つも二項対立があると機械的に見える。

### 改善

二項対立は1文書に1つまで。本当に対比が必要な場面だけ使う。

## 3つ組みの強制

### 問題例

```
成功の秘訣は3つあります。
1. 計画を立てる
2. 実行する
3. 振り返る
```

理由が2つなら2つでいい。5つなら5つでいい。無理に3つにまとめない。

## 機械的な列挙

「第一に」「第二に」「第三に」のような機械的な番号付けは避ける。

### 問題例

```
第一に、〇〇です。第二に、△△です。第三に、□□です。
これらを踏まえると、××という結論に至ります。
```

### 改善

番号付けを外すか、見出しで構造化する。

## 形式的な感情表現

ビジネス文書では感情表現自体を避ける。

### 問題例

```
とても感動しました。
心から嬉しく思います。
大変光栄に存じます。
```

### 改善

具体的なエピソードで感情を伝える。
「泣きそうになった」「3日間そのことしか考えられなかった」など、身体的・時間的な表現を使う。

## 改行の英語化

英語式の「1段落=1トピック」で空行を入れる書き方は避ける。

### 問題例

```
設定ファイルを開いてください。portの値を8080に変更します。保存したらサーバーを再起動してください。

再起動後、ブラウザでlocalhost:8080にアクセスします。ログイン画面が表示されれば成功です。
```

すべての改行が空行（段落分け）になっている。英語的。

### 改善

```
設定ファイルを開いてください。
portの値を8080に変更します。
保存したらサーバーを再起動してください。

再起動後、ブラウザでlocalhost:8080にアクセスします。
ログイン画面が表示されれば成功です。
```

### 日本語の改行ルール

改行の判断で最も重要なのは文脈。口に出して読んだとき、間を置かずに続けたい内容は改行しない。補足や言い換えなど、つなげて書いたほうが安心感のある文も同様。

| 場面 | 改行の仕方 |
|-----|----------|
| 間を置かずに続けたい内容 | 改行しない |
| 補足・言い換え | つなげて書く |
| 話題が変わる | 空行を入れて段落を分ける |
| 手順の説明 | 1ステップごとに改行してもよい |

### 見分け方

「すべての改行が空行になっている」「文が横に長く続いている」場合は英語式の改行を疑う。

## 書式の問題

### 太字の乱用

```
この機能は**非常に重要**です。**必ず**設定してください。
```

太字を使うと「重要だから目立たせたい」という意図が透けて見える。
文章の構造や言い回しで重要性を伝える。

### コロン終わりの見出し

```
## 設定方法:
## 注意事項:
```

コロンは英語圏の慣習。日本語では見出しに使わない。
ラベルには代わりに記号（■●※→【】など）を使う。

### 脈絡のない半角スペース

```
この 機能は 便利です。
```

英語の影響で単語間にスペースが入る。日本語では不要。

## チェックリスト

- 「重要です」「大切です」「必要です」が近くに3つ以上ないか
- すべての文が同じくらいの長さになっていないか
- 説明文中の箇条書きが4項目以上続いていないか
- 「本記事では」「以上のように」などの予告・要約文がないか
- 二項対立が2つ以上ないか
- 太字を使っていないか
- 見出しがコロン終わりになっていないか（ラベルは記号で代替）
- 謎の半角スペースがないか
- すべての改行が空行（段落分け）になっていないか
